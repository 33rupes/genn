Coding guidelines

Development workflow

GeNN development is done in a git repository on github. Continuous integration testing is provided by Jenkins, code coverage is measured with Codecov.

Coding conventions

GeNN is written in C++11 with the following coding conventions:

  • Use 4 spaces instead of tabs per indentation level
  • Class and namespace names are in UpperCamelCase
  • Variable names are in lowerCamelCase
  • Apart from in structs where they are public and accessed directly, member variables should have a m_ prefix and static member variables should have a s_ prefix.
  • All blocks, even those on a single line, should be enclosed in curly brackets.
  • While there is no fixed maximum line length, you should aim to make most lines less than around 100 characters so two files can be viewed side-by-side on a typical monitor.

Documentation

GeNN is documented using doxygen and all new methods should be documented using the following doxygen style: 

//! Brief description
/*! Any additional detailed description
    \tparam name description of template parameter
    \param name description of parameter (should begin with type)
    \return description of return value (should begin with type)*/

The Python code used to implement PyGeNN is documented using standard PyDoc blocks: 

"""Description
    
Args:
arg_name     --  description of argument (should begin with type)

Keyword args:
kwarg_name   -- description of keyword argument (should begin with type)

Returns description of return

Raises exception_type exception description (one per line)
"""

This is then converted to Doxygen using doxypypy. Documentation on additional features should also be added to the Best practices guide or User Manual as appropriate. Both of these pages document the usage of features from either Python or C++ and, to help with this, we have added some additional DoxyGen alises. Entire blocks of doxygen can be marked as being only applicable to one language as follows:

\add_toggle_cpp
C++-specific instructions
\end_toggle
\add_toggle_python
Python-specific instructions
\end_toggle

Code examples can be similarly marked:

\add_toggle_code_cpp
std::cout << "Hello" << std::endl;
\end_toggle_code
\add_toggle_code_python
print("Hello")
\end_toggle_code

and finally, within blocks of text, text can be marked like:

\add_cpp_python_text{C++,Python} is an \add_cpp_python_text{compiled,interpreted} language. This means \add_cpp_text{C++ is often faster}\add_python_text{Python can be used interactively}.

Sections marked in all of these ways can be switched between using the "Python" and "C++" buttons at the top of each page.

Logging

GeNN uses the plog logging library. It defines three logging channels Logging::CHANNEL_GENN, Logging::CHANNEL_CODE_GEN and Logging::CHANNEL_BACKEND which allow logging levels to be configured independently in different parts of GeNN. To log to these channels, GeNN provides the following shorthand logging macros:

  • LOGV_GENN
  • LOGD_GENN
  • LOGI_GENN
  • LOGW_GENN
  • LOGE_GENN
  • LOGF_GENN
  • LOGV_CODE_GEN
  • LOGD_CODE_GEN
  • LOGI_CODE_GEN
  • LOGW_CODE_GEN
  • LOGE_CODE_GEN
  • LOGF_CODE_GEN
  • LOGV_BACKEND
  • LOGD_BACKEND
  • LOGI_BACKEND
  • LOGW_BACKEND
  • LOGE_BACKEND
  • LOGF_BACKEND

where V, D, I, W, E and F refer to verbose, debug, information, warning, error and fatal log levels respectively. Within GeNN these macros are used like:

LOGD_BACKEND << "Backend debug information";

Model description

This is the part of GeNN which is exposed to the user, either directly if they are using C++ or via PyGeNN. It contains no backend-specific code and is written entirely in plain C++. The ModelSpec class is the 'entry-point' to this API and provides methods for setting model-level options such as simulation timestep (ModelSpec::setDT).

Some model properties such as the maximum axonal delay from a neuron group cannot be determined until the whole model is built i.e. all outgoing synapse groups have been added. Any

From this point onwards in the code generation process, the model is immutable and, as such, is passed to all subsequent stages as a const reference.

Kernel merging

Prior to GeNN 4.3.0, seperate update code was generated for each neuron and synapse group and these were fused together into large functions/kernels. However, for complex models, this resulted in extremely large kernels which took a long time to compile and performed poorly (see [3] for more details). To address this, the synapse, neuron and current source groups described previously are now 'merged' together once the model has been finalized. Merging is initiated by constructing a CodeGenerator::ModelSpecMerged object from a fully-initialised ModelSpecInternal.
